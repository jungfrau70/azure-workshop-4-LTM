# Azure Log Analytics & Azure Monitor Agent — Reference Guide

A comprehensive tutorial on data collection, configuration, cost optimization, and API integration using Azure Log Analytics Workspace (LAW) and Azure Monitor Agent (AMA).

The following five key topics provide a structured overview of the full pipeline.

---

### 1. Log Analytics Workspace (LAW) — Basics and Management

This section covers the concept and configuration of the Workspace, which forms the backbone of log monitoring.

* **Centralized data store:** LAW is the core infrastructure for collecting, analyzing, and monitoring all events and logs (Windows Event, Syslog, CEF, etc.) generated by devices, operating systems, and security applications (Microsoft Sentinel, Defender for Cloud, etc.) in a single location.
* **Workspace configuration and cost management:** There is no upfront creation cost. Billing follows a Pay-as-you-go model based on the volume of data ingested (GB) and the retention period. Setting a Daily Cap prevents unexpected cost spikes.
* **Table plans and data retention:** Ingested data is stored across multiple tables, each with a unique schema. Table plans are divided into `Analytics` (full KQL query support, higher cost) and `Basic` (restricted to simple queries — no joins, lower cost). Retention periods can be configured as Interactive (available for immediate querying) and Archival (long-term, low-cost storage).

### 2. Azure Monitor Agent (AMA) and Data Collection Rules (DCR)

A new agent and rule system introduced to overcome the limitations of the legacy agent (MMA).

* **AMA advantages:** The legacy MMA approach used a Workspace ID and key for authentication, which was a security weakness, and forced the same configuration on all machines. AMA authenticates with Managed Identity and provides far more flexible, independent configuration through Data Collection Rules (DCR).
* **Multi-homing:** With AMA, performance logs from a single machine can be sent to a "performance monitoring Workspace" while security logs go to a separate "Sentinel Workspace."
* **Role of DCR:** A DCR explicitly defines which sources (resources) to collect data from, what data (data sources) to collect, how to transform it (Transform), and where to send it (destination). It is the central configuration resource for AMA-based collection.

### 3. AMA Deployment and Custom Collection per Server Platform

Practical configuration for installing the agent on real Windows and Linux servers and collecting logs.

* **Windows server collection:**
  * Collects performance counters and event logs (System, Application, Security) by default.
  * Connecting on-premises machines via Azure Arc and assigning a DCR automatically deploys AMA as an extension.
* **Custom logs:** To collect logs from only specific components — such as a DNS server or Windows Hello — beyond the default options, extract an XPath query in XML format from Windows Event Viewer and manually add it to the DCR for precise, targeted collection.

* **Linux server collection:**
  * AMA for Linux does **not** require Python (Python was a prerequisite for the legacy MMA/OMS agent only; AMA has no such dependency).
  * Syslog facility and log level (Debug, Warning, etc.) are specified for collection. Once configured, data appearing in the `Syslog` and `Perf` tables can be confirmed via KQL.

### 4. Collection Cost Optimization (Data Filtering and Transformation)

Advanced techniques for dramatically reducing unnecessary costs that arise when data is collected without filtering.

* **Bill Size analysis:** Use the `_BilledSize` and `_IsBillable` properties in KQL queries to determine the byte size of individual log records and whether they are billable. The `Usage` table tracks ingestion volume (MB) per hour and per table.
* **ETL-based table transformation:** A strategy for **reducing data size per event** by dropping unnecessary columns or empty field values (e.g., Tenant ID, Device Version that appear identically on every log record) at the ingestion stage — before data is written to the table.
  * Use the "Table Transformation Editor" in the Azure Portal, or add a `project-away` query to the `transformKql` section of the DCR JSON structure to cleanse data. This can reduce total data ingestion by more than 25%.

### 5. API Integration and Operations Automation (Advanced)

Methodology for using APIs and scripts rather than a GUI to manage large-scale enterprise environments.

* **DCR access via REST API:** Create an App Registration in Microsoft Entra ID (formerly Azure AD), obtain an OAuth 2.0-based token, and use Postman or PowerShell to query and configure DCR resources.
* **Log export via Log Analytics API:**
  * Call a predefined KQL query via API to extract raw data from the workspace.
  * Beyond simple extraction with Postman and PowerShell, demonstrates how to build a pipeline that **combines Python's Pandas (DataFrame) with the Microsoft Graph API** to automatically send extracted logs as a scheduled email report.
